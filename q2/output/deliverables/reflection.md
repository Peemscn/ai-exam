# Reflection — Gacha Drop Rate Simulator

งานนี้ผมวาง business flow กับ requirement ให้ชัดก่อน แล้วใช้ Claude (ผ่าน Claude Code) เป็นมือเขียน — ให้มันอ่านโจทย์ทุก tab สรุปสิ่งที่ต้องทำ ออกแบบ แล้วลงมือเขียน `gacha-simulator.html` ทั้งไฟล์ ส่วนผมทำหน้าที่คุมทิศทาง ตัดสินใจว่าจะเอา core + bonus ให้ครบ และคอยตรวจผลลัพธ์ทุกขั้น

ที่ผมพอใจที่สุดคือตัว roll logic เพราะมันสุ่มตาม cumulative rate จริง ๆ ไม่ได้ fix ผลไว้ ลองตั้ง rate 1/9/30/60 แล้วสุ่มเป็นหมื่นรอบ สัดส่วนที่ออกมาก็ใกล้เคียงตลอด ส่วน Player POV ที่เป็น Monte Carlo กับกติกา free roll (สุ่มครบ X ครั้งได้ฟรี Y ครั้ง โดยฟรีไม่งอกฟรีต่อ) ก็คำนวณตรง แล้วก็ดีใจที่ทำออกมาเป็นไฟล์เดียวเปิด `file://` ได้เลย ไม่ต้องพึ่ง server หรือ CDN — กราฟวาดเองด้วย CSS, ใส่ dark mode กับ responsive ครบ, ส่วน bonus (pity, history, chart) ก็ทำหมด

รอบนี้บอกตรง ๆ ว่าไม่ค่อยมีอะไรติดขัด ส่วนหนึ่งเพราะผมวาง logic เป็นฟังก์ชันแยก (`pickRarity` / `rollOnce` / `runMC`) ตั้งแต่แรก พอจะต่อ feature หรือเขียนเทสก็ง่าย แล้วระหว่างทางก็มี unit test กับ Playwright คอยจับอยู่ เลยไม่มีปัญหาค้างตอนท้าย ถามว่า AI เขียนผิดไหม — รอบนี้ไม่เจอ logic ที่ผิดเลย และที่กล้าพูดแบบนี้เพราะผมให้เขียน unit test ครอบ logic หลักไว้ (rate รวม 100, free-roll, pity, การกระจายของการสุ่ม, pool) ผ่าน 12/12 ทั้ง `node` และ `bun` บวกกับเปิดเบราว์เซอร์จริงไล่ verify ทุก feature ด้วย Playwright ถ้ามีอะไรเพี้ยน test หรือการรันจริงมันจะฟ้องก่อน เลยไม่ปล่อยให้ของผิดหลุดไป

ครั้งหน้าผมจะคุม business flow กับ requirement ให้แม่นตั้งแต่ตอนอ่านโจทย์ วาง state แล้วแยกฟังก์ชันตั้งแต่ต้น (อันนี้ทำแล้วเวิร์กจะทำต่อ) และเขียน test คู่ไปกับ feature เสมอ ถ้าไม่ผ่านก็แก้จนกว่าจะผ่าน
